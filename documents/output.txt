STACKRADAR Real-Time Tech Intelligence Engine Market Analysis & Product Validation Is StackRadar viable? Who needs it, and how will they use it? v1.0  |  May 2026 Confidential — Internal Use 1. Executive Summary StackRadar is positioned at the intersection of developer education, tech intelligence, and real-time data aggregation. This document evaluates the product's market viability, primary user segments, go-to-market considerations, and competitive landscape. The conclusion: the market opportunity is real, but the product must graduate from a domain-level dashboard to a tool-level intelligence engine to deliver genuine value. 2. Problem Statement The developer ecosystem evolves at a pace that no individual can track manually. A student choosing between React and Svelte, an engineering manager deciding whether to adopt Rust, or a recruiter calibrating their job requirements — all face the same problem: reliable, real-time signal about what the industry is actually using and trending toward. Current alternatives fall into two categories: Surveys (Stack Overflow Developer Survey, JetBrains State of Developer Ecosystem): Annual cadence, lag of 12–18 months, no real-time signal. Social media / GitHub Trending: High noise, no scoring or aggregation, no historical context. Analyst reports (ThoughtWorks Tech Radar, TIOBE): Designed for enterprise architects, not students or mid-career developers. StackRadar's core thesis is correct: there is a gap for a real-time, developer-centric intelligence tool. The risk is that the current implementation does not yet close that gap. 3. Target User Segments 3.1 Primary: Students & Early-Career Developers This is the highest-volume, most underserved segment. Students face a fundamental learning prioritization problem: what framework or language should I invest 100+ hours into? They have limited time, limited professional networks, and high sensitivity to wasted effort. Attribute Detail Age 18–28 Job Status Students, bootcamp graduates, first/second job developers Key Pain 'Which tech should I learn next?' — No trusted, real-time answer exists StackRadar Value Ranked tool scores + momentum trends + suggested learning paths Acquisition Channel Reddit (r/learnprogramming, r/cscareerquestions), Twitter/X, dev Discord servers 3.2 Secondary: Mid-Career Developers & Tech Leads Developers with 3–10 years of experience regularly evaluate new tools for adoption in side projects, greenfield features, or to stay competitive in job interviews. They are time-constrained and value data density over hand-holding. Willing to pay for a premium tier if the data depth is demonstrably superior to free alternatives. Likely to share StackRadar links in team Slack channels, creating organic B2B virality. Will compare StackRadar data against their own GitHub/LinkedIn observations — data accuracy is critical for retention. 3.3 Tertiary: Engineering Managers & CTOs This segment uses tech intelligence for hiring strategy, technology roadmap decisions, and team upskilling. They are low volume but high intent. A 'team dashboard' or API access tier could unlock this segment in a later phase. 3.4 Adjacent: Recruiters & HR Tech Tech recruiters building job descriptions need to know which skills are trending. This segment is monetizable via B2B SaaS but should not be the launch focus — it requires a different UX, compliance considerations, and sales motion. 4. A freemium model with 2% conversion is achievable within 18 months of a polished launch. 5. Competitive Landscape Competitor Real-Time Tool-Level Free Tier Gap StackRadar Fills StackOverflow Survey ❌ Annual ✅ ✅ Real-time momentum, not year-old snapshots GitHub Trending ✅ Daily ✅ ✅ No scoring, no sentiment, no aggregation ThoughtWorks Radar ❌ Quarterly ✅ ✅ Developer-first UX, not enterprise TIOBE Index ✅ Monthly ✅ Languages only ✅ Framework-level; multi-source aggregation dev.to / HN ✅ ❌ No scoring ✅ Structured score + trend lines + roadmap tie-in StackRadar's defensible moat is the combination of multi-source real-time aggregation + AI sentiment + actionable learning roadmaps. No single competitor does all three. The product must achieve all three to justify its existence. 6. User Journey & Usage Patterns 6.1 Primary Usage Flow (Student) Student hears about a technology (e.g., 'Should I learn Rust?') on Reddit or YouTube. Visits StackRadar, searches for 'Rust' in the Explore page. Sees a Momentum Score (e.g., 78/100), trend line (up 12% last month), sentiment breakdown (GitHub stars, HN mentions, Dev.to posts). Clicks 'Roadmap' — sees a curated learning path for Rust (beginner → systems → async). Bookmarks the page / creates a free account to save their tracked tools. Returns weekly to check score momentum as they learn. 6.2 Secondary Usage Flow (Tech Lead) Engineering team is evaluating moving from REST to GraphQL. Tech lead opens StackRadar 'Compare' — enters GraphQL vs REST vs gRPC. Views side-by-side momentum scores, GitHub contribution trends, and community sentiment. Shares the comparison link in a team Slack message. Returns when a new alternative emerges (e.g., tRPC) to add to comparison. 7. Product-Market Fit Assessment PMF Signal Current Status What Must Change Real problem exists ✅ Confirmed Nothing — the problem is validated Real data delivered ⚠️ Partial Trends page real; Explore & Roadmaps are hardcoded Tool-level (not domain) insight ❌ Missing Scoring engine must track React, Rust, Deno — not 'WebDev' Historical trend charts ❌ Missing growth: [] arrays must be populated — time-series DB needed Actionable output (roadmaps) ⚠️ Hardcoded Dynamic roadmaps served from backend Retention mechanism ❌ None yet Saved tools / watchlist / weekly email digest 8. Key Risks Data Quality Risk: If GitHub API rate limits degrade data freshness, the core value proposition collapses. Mitigation: authenticated GraphQL + caching layer + fallback to last-known-good score. Competitor Entry Risk: A well-funded tool (e.g., built on top of GitHub Octoverse data) could replicate the scoring model quickly. Mitigation: roadmap integration + user accounts create switching costs. Trust Risk: If users catch hardcoded data, they will dismiss the entire platform. Mitigation: remove all hardcoded content before any public launch. Monetization Timeline: The free tier must be compelling enough to drive word-of-mouth before a paid tier is introduced. Do not paywall real-time data in V1. 9. Recommendation StackRadar is a viable product in a real market. It is NOT production-ready today. The core architecture is sound and the concept is differentiated. The critical path to PMF is: (1) tool-level scoring, (2) real time-series charts, (3) zero hardcoded data. With those three changes, StackRadar can realistically reach 50,000 MAU within 12 months of launch.

STACKRADAR Real-Time Tech Intelligence Engine Product Requirements Document StackRadar — Production V1.0 Feature Specification v1.0  |  May 2026 Confidential — Internal Use 1. Product Vision StackRadar is a real-time technology intelligence engine that helps developers at every stage of their career understand what tools are rising, falling, or dominating in the industry — and then gives them a clear path to learn those tools. 2. Scope In Scope (V1.0) Real-time tool-level scoring dashboard (Trends page) Dynamic Explore page (Technology Map) — no hardcoded data Dynamic Roadmaps fetched from backend, editable by admin Tool comparison page (side-by-side metrics) Historical trend charts (30/90/365 day views) User accounts with saved/watched tools Public REST API (read-only) for developer integrations Out of Scope (V1.0) B2B/enterprise team dashboards Mobile native app (PWA acceptable) User-submitted tool reviews or ratings Job board or recruiter features 3. Feature Requirements FR-01: Real-Time Scoring Engine The scoring engine must produce a 0–100 score for each individual tool (not just domains). The score is a weighted composite of the signals below. Signal Source Weight Metric Captured Update Frequency GitHub 35% Stars, Forks, Issues opened (30d) Every 60 min HackerNews 25% Ask HN / Show HN mentions, points Every 30 min Dev.to 20% Posts published, reactions, views Every 30 min Reddit 15% Post count, upvotes (relevant subs) Every 30 min RSS Tech News 5% Article mentions, domain authority Every 2 hours Acceptance Criteria Score for each tool updates at least every 60 minutes during business hours. Score must reflect actual tool-level repositories, not broad topic searches. Score delta (7-day change) displayed as +/- percentage with directional arrow. Sentiment polarity (positive / neutral / negative) derived via Groq/LLM batch job — not naive substring matching. FR-02: Historical Time-Series Charts Every tool detail page must display a line chart of its score over time. The chart must support three time-range toggles: 30 days, 90 days, and 365 days. Data Model Requirement A new Snapshot table must be written by the scheduler on every successful scraping run. Minimum fields: tool_id (FK) recorded_at (timestamp, indexed) score (float) github_stars_delta (int) mention_count (int) sentiment_score (float, -1.0 to 1.0) Acceptance Criteria Recharts LineChart renders with real data points — no empty arrays. Missing days shown as interpolated gaps, not broken lines. API endpoint /api/v1/tools/{tool_id}/history?days=30 returns sorted JSON array. FR-03: Dynamic Explore Page The Explore (Technology Map) page must be fully driven by backend data. No TypeScript hardcoded arrays. The languages.ts and roadmaps.ts static files must be deleted or replaced with API calls. Backend Requirement New endpoint: GET /api/v1/tools — returns paginated list of all tracked tools with score, domain, and tag. New endpoint: GET /api/v1/domains — returns all domains with their top 5 tools. Admin endpoint: POST /api/v1/admin/tools — allows adding new tools to tracking (auth-protected). Acceptance Criteria Adding a new tool via admin API causes it to appear on the Explore page within the next scraping cycle. No hardcoded tool names in any frontend TypeScript file. FR-04: Dynamic Roadmaps Learning roadmaps must be database-driven and served from the backend. Each roadmap is a structured JSON object defining stages, resources, and tool recommendations. Data Model Roadmap table: id, tool_id (nullable for multi-tool roadmaps), title, description, stages (JSON array), updated_at Stage schema: { order: int, title: string, description: string, resources: [{ url, title, type }] } Acceptance Criteria Roadmap content editable via admin API — no code deployment required to update content. Roadmap page loads content from /api/v1/roadmaps/{id} with <500ms P95 response time. FR-05: Tool Comparison Allow users to compare 2–4 tools side by side. Comparison must include: current score, 30-day trend, top signal source, domain category, and sentiment breakdown. Acceptance Criteria Comparison URL is shareable (state encoded in query params). All comparison data fetched from backend — no frontend-side calculations. FR-06: User Accounts & Watchlist Authentication via Clerk (already integrated). Users can save tools to a personal watchlist. Watchlist displays on dashboard with delta changes since last visit. Acceptance Criteria Watchlist persists across sessions. Weekly email digest (opt-in) summarizes score movements for watched tools. 4. STACKRADAR Real-Time Tech Intelligence Engine Technical Task Breakdown Actionable sprint-ready engineering tasks for production readiness v1.0  |  May 2026 Confidential — Internal Use Overview This document breaks the StackRadar production upgrade into 5 phases, each with atomic, assignable engineering tasks. Tasks are ordered by dependency — Phase 1 must be completed before Phase 2, and so on. Each task includes an effort estimate, the files affected, and explicit acceptance criteria. Priority Key: 🔴 Critical (blocks launch) | 🟡 High (degrades core value) | 🟢 Medium (improves quality) | ⚪ Low (polish) Phase 1: Data Pipeline Hardening TASK-001 🔴 — Fix GitHub Scraper Rate Limiting File: backend/app/services/scraper.py Problem Current implementation uses unauthenticated /search/repositories with 1-2s sleep. Results in 403 errors within minutes of startup, silently halting data collection. Solution Load GITHUB_TOKEN from environment via Pydantic settings. Add Authorization: Bearer {token} header to all GitHub requests. Replace /search/repositories with targeted GraphQL queries for specific repos (e.g., query for facebook/react, not 'react' topic). Implement exponential backoff: on 403, wait 60s, retry 3x, then skip and log. Add X-RateLimit-Remaining header check — pause if remaining < 10. Acceptance Criteria 24-hour run with valid GITHUB_TOKEN produces zero 403 errors in logs. At least 30 tools have GitHub data populated after one full scraping cycle. Estimate: 4–6 hours TASK-002 🔴 — Shift from Domain to Tool-Level Scoring File: backend/app/services/scoring.py, backend/app/models/ Problem classify_text_to_domains() does naive substring matching assigning content to broad domains. No individual tool scores exist. React, Vue, Angular all map to 'WebDev'. Solution Create a TOOL_REGISTRY dict: { 'react': { repo: 'facebook/react', keywords: ['react', 'reactjs'], domain: 'frontend' }, ... } for 40+ tools. Rewrite scoring to iterate over TOOL_REGISTRY, building per-tool metrics. Ensure Tool model has individual score, github_stars, mention_count, sentiment_score fields. Migrate existing broad domain scores to a separate DomainAggregate table (preserves Trends page). Acceptance Criteria GET /api/v1/tools returns 40+ tools each with a non-zero score. React and Angular have distinct, different scores. Estimate: 8–12 hours TASK-003 🔴 — Implement Time-Series Snapshots Files: backend/app/models/, backend/alembic/, backend/app/services/scheduler.py Problem No historical data is stored. growth: [] returns empty from all endpoints. Charts are empty. Solution Create ToolSnapshot table: id, tool_id (FK), recorded_at (indexed), score, github_stars_delta, mention_count, sentiment_score. Write Alembic migration for new table. In scheduler.py run_scraper_loop(), after scoring, call save_snapshot() for each tool. New endpoint: GET /api/v1/tools/{id}/history?days=30 — queries ToolSnapshot, returns sorted list. Acceptance Criteria After 48 hours of runtime, history endpoint returns at least 48 data points per tool. Frontend Recharts LineChart renders populated lines. Estimate: 6–8 hours TASK-004 🟡 — LLM Sentiment Analysis (Groq) Files: backend/app/services/scoring.py, backend/app/core/config.py Problem Sentiment is currently naive substring matching. Negative articles about a technology count as positive mentions. Solution Collect up to 20 text snippets per tool per scraping cycle. Batch them into a single Groq API call with prompt: 'For each text, return sentiment: positive|neutral|negative for the technology mentioned.' Parse JSON response, store average sentiment_score (-1.0 to 1.0) in ToolSnapshot. Add GROQ_API_KEY to Pydantic settings with graceful fallback to neutral (0.0) if key absent. Acceptance Criteria An article titled 'Why React is dead' scores negative sentiment for React. Groq API errors do not crash the scraper loop — logged and skipped. Estimate: 4–6 hours Phase 2: Frontend Data Decoupling TASK-005 🔴 — Remove Hardcoded Explore Data Files: frontend/src/app/explore/page.tsx, frontend/src/data/languages.ts Solution Delete frontend/src/data/languages.ts (or archive to legacy/). Replace static data with fetch to GET /api/v1/tools and GET /api/v1/domains. Implement loading skeleton while data fetches. Implement error boundary with graceful fallback message. Acceptance Criteria New tool added via admin API appears on Explore page after next scraping cycle without code change. No TypeScript array containing tool names exists in frontend. Estimate: 4–6 hours TASK-006 🔴 — Remove Hardcoded Roadmap Data Files: frontend/src/app/roadmaps/page.tsx, frontend/src/data/roadmaps.ts, backend/ Solution Create Roadmap and RoadmapStage database models with Alembic migration. Write seed script to import existing hardcoded roadmaps into DB (one-time). Create GET /api/v1/roadmaps and GET /api/v1/roadmaps/{id} endpoints. Frontend fetches roadmaps dynamically — remove roadmaps.ts. Estimate: 8–10 hours TASK-007 🟡 — Populate Historical Charts in Frontend Files: frontend/src/app/tools/[id]/page.tsx Solution Replace growth: [] static array with fetch to /api/v1/tools/{id}/history. Add 30d / 90d / 365d toggle buttons — pass ?days= param. Handle empty state (< 7 data points) with 'Collecting data — check back soon' message. Estimate: 3–4 hours Phase 3: Security & Production Hygiene TASK-008 🔴 — Secrets & Environment Audit Files: docker-compose.yml, backend/app/core/config.py, .github/workflows/ Checklist Audit all .env.example files — ensure no real keys are committed. Enable GitHub secret scanning on repository settings. Move all API keys (GITHUB_TOKEN, GROQ_API_KEY, DATABASE_URL) to Kubernetes Secrets / Railway/Render environment variables. Ensure Pydantic Settings model raises ValidationError on startup if required env vars are absent. Add .env to .gitignore (verify it is already present). Estimate: 2–3 hours TASK-009 🟡 — Rate Limiting & CORS Hardening Files: backend/app/main.py Solution Add slowapi (or fastapi-limiter) middleware: 60 req/min per IP on public API endpoints. Restrict CORS origins to production frontend domain (not *) in production environment. Add Helmet-equivalent security headers via fastapi middleware. Estimate: 2–3 hours TASK-010 🟡 — Database Connection Pooling Files: backend/app/db/session.py Solution Configure SQLAlchemy pool_size=10, max_overflow=20, pool_timeout=30. Add pool_pre_ping=True to handle stale connections. Ensure Alembic migrations run on startup with proper error handling. Estimate: 1–2 hours Phase 4: Infrastructure & Observability TASK-011 🟡 — Structured Logging Files: backend/app/main.py, backend/app/services/ Solution Replace print() and basic logging with structlog or loguru. Every scraping cycle logs: tools_processed, errors, duration_ms, data_points_written. Log format: JSON for production (parsed by CloudWatch/Datadog), human-readable for development. Estimate: 2–3 hours TASK-012 🟡 — Health & Readiness Endpoints Files: backend/app/api/ Solution GET /health — returns 200 with { status: 'ok', db: 'connected', last_scrape: '<iso timestamp>' }. GET /ready — returns 503 until first scraping cycle completes. Configure Kubernetes liveness/readiness probes to use these endpoints. Estimate: 1–2 hours TASK-013 🟢 — CI/CD Pipeline (GitHub Actions) Files: .github/workflows/ Solution Backend workflow: run pytest, mypy type check, build Docker image, push to GHCR on merge to main. Frontend workflow: run next build, run vitest, push Docker image on merge to main. Deployment workflow: kubectl rollout restart after image push (or ArgoCD sync). Estimate: 4–5 hours Phase 5: UX & Retention TASK-014 🟢 — User Watchlist Requires Clerk auth (already integrated). Users can track specific tools. Backend: Watchlist table (user_id, tool_id, added_at). CRUD endpoints under /api/v1/users/watchlist. Frontend: Heart/bookmark icon on each tool card. Watchlist tab on user profile. Show delta since last visit: 'React +5pts since you last visited.' Estimate: 6–8 hours TASK-015 🟢 — Weekly Email Digest Use Resend or SendGrid for transactional email. Background job runs every Monday at 09:00 UTC. Email template: top 5 movers in user's watchlist + top 3 rising tools globally. Unsubscribe link with one-click opt-out stored in DB. Estimate: 5–6 hours TASK-016 ⚪ — SEO & Open Graph Each tool page has unique <title>, <meta description>, and og:image (auto-generated score card). robots.txt, sitemap.xml auto-generated from tool list. Next.js generateMetadata() used for all dynamic routes. Estimate: 3–4 hours Phase Summary & Total Estimate Phase Focus Est.
